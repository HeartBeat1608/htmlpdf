System Design Review

Distributed Cache Layer — Q2 2026

Prepared by: Platform Engineering · Reviewed by: Architecture Board · Status: Draft

1. Executive Summary

This document reviews the proposed architecture for a distributed cache layer to be introduced between the application tier and the primary PostgreSQL cluster. The primary goals are to reduce database read load by 70%, decrease p99 read latency from 180ms to under 20ms, and provide a horizontally scalable read path for the product catalogue and user session services.

Cache invalidation and naming things are the two hardest problems in computer science. The architecture below addresses the former with a write-through strategy and explicit TTL discipline; the latter remains, as always, the team's problem.

2. Current State

The existing architecture routes all read traffic directly to the primary database replica set. Under peak load (08:00–10:00 and 17:00–19:00 PST), the replica cluster sustains approximately 42,000 queries per second, of which profiling shows 68% are identical repeated lookups against four hot tables: products, categories, user_sessions, and feature_flags.

2.1 Observed Failure Modes

  • Read replica lag exceeds 2 seconds during peak batch import jobs, causing stale reads in the product search service
  • Connection pool exhaustion on the catalogue service (max_connections=500 reached at ~38k QPS)
  • Cold-start latency spike of 800ms when deploying new catalogue service instances — each instance primes its local LRU independently

3. Proposed Architecture

The proposed solution introduces a two-tier cache using Redis Cluster 7.2 as the primary shared cache and an in-process LRU (bounded at 256MB per instance) as the L1 tier. The write path remains synchronous to PostgreSQL; cache population uses a write-through strategy on mutation and a read-through fallback on cache miss.

3.1 Key Design Decisions

  1. Write-through on all mutations — eliminates the stale-read window present in lazy invalidation approaches. The additional write latency (estimated +4ms p50) is acceptable given the read improvement.
  2. TTL of 300s for product/category data — product data changes infrequently; a 5-minute TTL provides a safety net against cache poisoning without meaningful staleness risk.
  3. No TTL for feature flags — flags are explicitly invalidated on write via a pub/sub channel. Allowing stale flags to persist would create inconsistent feature rollout behaviour.
  4. Session data: TTL mirrors session expiry — cache TTL is set equal to the session token expiry time, so cache and database age out together.

4. Implementation Notes

4.1 Cache Key Schema

All cache keys follow a namespaced hierarchical format to allow targeted invalidation:

{service}:{entity}:{id}[:{variant}]

Examples:
  catalogue:product:48291
  catalogue:category:112:tree
  sessions:user:9938821
  flags:global:dark_mode_rollout

4.2 Go Client Wrapper

type CacheClient struct {
    l1  *lru.Cache        // in-process, 256MB cap
    l2  *redis.ClusterClient
    ttl map[string]time.Duration
}

func (c *CacheClient) Get(ctx context.Context, key string) ([]byte, error) {
    if v, ok := c.l1.Get(key); ok {
        return v.([]byte), nil
    }
    v, err := c.l2.Get(ctx, key).Bytes()
    if err == nil {
        c.l1.Add(key, v)
    }
    return v, err
}

5. Risk Assessment

Risk Likelihood Impact Mitigation
Cache stampede on cold start Medium High Probabilistic early expiration (PER) + request coalescing via singleflight
Redis cluster partition Low High Circuit breaker falls through to database; degraded but functional
Memory pressure on L1 Medium Low LRU eviction bounded at 256MB; monitor with runtime/metrics
Stale data after failed write-through Low Medium Transactional outbox pattern ensures eventual cache consistency

6. Acceptance Criteria

  • Cache hit rate ≥ 85% within 30 minutes of warm-up under production load
  • p99 read latency for cached entities ≤ 20ms (measured at the service boundary)
  • Zero increase in error rate during Redis cluster failover (circuit breaker must activate within 500ms)
  • No stale reads persisting beyond max(TTL, explicit invalidation) + 1s
